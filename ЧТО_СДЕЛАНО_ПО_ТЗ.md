# Что сделано по ТЗ v5.0

Краткий список реализованного по чек-листу и плану.

## База данных (Prisma)

- Добавлены таблицы: **Manager**, **HandoffAssignment**, **StopList**, **WhatsAppAccount**, **Followup**, **OffHoursQueue**, **SystemConfig**, **Prompt**, **KnowledgeDocument**, **KnowledgeFragment**.
- В **LeadSession** добавлены поля: phoneMasked, waAccountId, slaNotified, zoom-поля, followupStopped и др.

**Важно:** после пулла выполни миграцию:
```bash
npx prisma migrate dev --name add_tz_tables
```

## Стоп-лист

- Проверка номера в стоп-листе **перед созданием** lead_session. Если в стоп-листе — сессия создаётся со статусом INIT_FAILED, в очередь не ставится.
- Нормализация телефона при создании лида.
- **Admin API:** `GET/POST/DELETE /api/admin/stop-list` — список, добавление, удаление.

## amoCRM Chats API (WhatCRM)

- **AmoChatsService:** создание чата (`createChat`), отправка сообщения (`sendMessage`).
- При отправке init-сообщения, если в .env заданы `AMOCRM_SUBDOMAIN` и `AMOCRM_ACCESS_TOKEN`, сообщение уходит в amoCRM Chats API и сохраняется `conversation_id` и `external_message_id`.

## LLM (OpenAI)

- **LlmService:** вызов OpenAI Chat Completions (модель gpt-4o-mini), ограничение ответа 400 символов.
- В **DialogService:** если задан `OPENAI_API_KEY`, ответ генерируется через LLM с системным промптом по этапу диалога; иначе — шаблонные ответы по статусу.

## Рабочее время

- **AppConfigService:** чтение TIMEZONE, SEND_WINDOW_START, SEND_WINDOW_END, WORK_DAYS из .env. Метод `isWithinWorkingHours()`. Пока не подключён к init queue (можно доработать run_after по следующему рабочему окну).

## Follow-up

- Таблица **Followup**, статусы SCHEDULED / SENT / CANCELLED / FAILED.
- **FollowupService:** планирование шага 1/2/3, отмена по сессии (`cancelForSession`).
- После отправки init-сообщения автоматически планируется **follow-up №1** через 24 часа.
- При **входящем сообщении** клиента все запланированные follow-up по этой сессии отменяются.
- **FollowupScheduler** (cron раз в минуту): обрабатывает записи с dueAt <= now, пишет сообщение в Message и обновляет статус сессии (FOLLOWUP_1/2/3).

## Calendly

- **CalendlyService:** метод `getAvailableTimes(eventTypeUuid, startTime, endTime)`. При наличии `CALENDLY_API_TOKEN` возвращает слоты из Calendly API.

## RAG

- **RagService:** заглушка поиска по **KnowledgeFragment** (без pgvector/embedding). Метод `search()`, `shouldEscalate(scores)` при низком confidence.

## Admin API

- **GET /api/admin/system-config** — чтение из таблицы SystemConfig (если пусто — дефолтные значения).
- **GET/POST/PUT/DELETE /api/admin/managers** — CRUD менеджеров (amoUserId, name, telegramId, whatsappPhone, active, excludeFromRotation, weight).
- **GET/POST/DELETE /api/admin/stop-list** — стоп-лист.

## Эскалация

- **EscalationService:** при вызове `handoff(leadSessionId, reason)` выбирается менеджер по round-robin (active, не excludeFromRotation), создаётся запись в **HandoffAssignment**, сессия переводится в HANDOFF_TO_HUMAN с handoffAt и assignedManagerId.
- **Admin API:** `POST /api/admin/lead-sessions/:id/handoff` с телом `{ "reason": "..." }` — ручная эскалация для теста.
- Создание задачи в amoCRM при handoff — в доработке (нужен вызов amoCRM API задач).
