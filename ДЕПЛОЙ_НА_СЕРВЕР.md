# Деплой на сервер заказчика

Заказчик разворачивает проект **только через GitHub** (по SSH, без прямого захода на сервер). Файлы на сервер попадают через `git clone` / `git pull`.

---

## Важно: пароли и ключи

**Не коммитить в Git:** пароли БД, SSH, FTP, JWT_SECRET, ключи API.  
Они хранятся только в **.env на сервере** (файл .env в .gitignore и в репозиторий не попадает).

---

## Что нужно на сервере для нашего проекта

Текущий стек приложения:
- **Node.js** (LTS)
- **PostgreSQL** (не MySQL) — база для Prisma
- **Redis** — очередь (init_queue, follow-up)

У заказчика по умолчанию указан **MySQL**. Для этого бота нужны именно **PostgreSQL и Redis**. Варианты:

1. **Попросить у хостинга (Webdock)** установить PostgreSQL и Redis на этот же сервер (или выдать отдельные сервисы).
2. **Или** поднять их в **Docker** на этом же VPS (если есть права на установку Docker), тогда в репозитории уже есть `docker-compose.yml`.

Если оставить только MySQL — приложение **не заработает** без доработок (Prisma перевести на MySQL и править схему).

## Как заказчику деплоить после пуша в GitHub

У кого есть SSH-доступ к серверу (логин `admin` по ключу):

### Первый раз (клонирование)

```bash
cd /var/www
sudo git clone https://github.com/mirosnikovevpatij-design/ai-sales-bot.git
cd ai-sales-bot
```

Создать на сервере файл **.env** (не из Git):

```bash
cp .env.example .env
nano .env   # или vi — вписать DATABASE_URL, REDIS_*, ключи API
```

Подставить в .env:
- `DATABASE_URL` — строка подключения к **PostgreSQL** на этом сервере (хост, пользователь, пароль, база).
- `REDIS_HOST`, `REDIS_PORT` — если Redis на этом же сервере: `localhost` и `6379`.
- Остальное по **КЛЮЧИ_И_НАСТРОЙКИ.md**.

Установка и запуск (если на сервере есть Node и PM2 или аналог):

```bash
npm install
npx prisma generate
npx prisma migrate deploy
npm run build
# Запуск в фоне, например:
# PM2: pm2 start dist/main.js --name ai-sales-bot
# или systemd (см. ниже)
```

Если используют **Docker**:

```bash
docker compose up -d
# Приложение тогда должно быть в контейнере (нужен Dockerfile в репо или запуск node из хоста при поднятых postgres+redis в docker)
```

### Дальше (после каждого пуша в GitHub)

Тот, у кого SSH, заходит на сервер и тянет обновления:

```bash
cd /var/www/ai-sales-bot
git pull origin main
npm install
npx prisma migrate deploy
npm run build
pm2 restart ai-sales-bot
```

(или перезапуск своего сервиса, если не PM2).

## Домен и веб

Сайт заказчика: **https://chatbot1.vps.webdock.cloud/**  
Корень веб-сервера: **/var/www/html**.

Наше приложение — **API (NestJS)** на порту 3000. Нужно в nginx (или аналоге) настроить проксирование, например:

- `https://chatbot1.vps.webdock.cloud/api` → `http://127.0.0.1:3000/api`

Тогда вебхуки amoCRM/WhatCRM будут указывать на `https://chatbot1.vps.webdock.cloud/api/webhooks/...`.  
Пока вебхуки не оплачены, этот же URL можно использовать для ручной проверки (см. **ПОДГОТОВКА_БЕЗ_API.md**).

Админку (React) можно собрать (`cd admin-frontend && npm run build`) и выложить в /var/www/html или в подпапку, чтобы открывать в браузере.

## Кратко для передачи заказчику

1. На сервере нужны **PostgreSQL и Redis** (не только MySQL). Запросить у хостинга или поднять через Docker.
2. Клонировать репозиторий из GitHub в `/var/www/ai-sales-bot`, создать на сервере `.env` с ключами (см. .env.example и КЛЮЧИ_И_НАСТРОЙКИ.md).
3. Деплой после пуша: `git pull`, `npm install`, `npx prisma migrate deploy`, `npm run build`, перезапуск приложения (PM2/systemd).
4. Настроить nginx (или аналог), чтобы запросы к `/api` шли на порт 3000.
5. Пароли и секреты хранить только в .env на сервере, в Git не коммитить.
